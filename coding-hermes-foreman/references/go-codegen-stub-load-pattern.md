# Stub Load Function Pattern for Code-Generated Dependencies

When a Go package depends on code generated by an external tool (bpf2go, protoc, stringer, etc.), the package won't compile until `go generate` has been run. This blocks development, CI, and testing on machines without the toolchain.

**The pattern:** use a package-level `var` (not `func`) as the stub for the generated load function. The Go compiler resolves type references at compile time but the stub is only called at runtime. The package compiles cleanly; the runtime error is caught by the caller as a graceful degradation path.

## Why `var` not `func`

A `func` stub must match the generated function's signature exactly. If the generated function signature changes (different parameter names, return types), the stub goes out of sync and must be updated. A `var` holding an anonymous function is more flexible — Go only checks that the assignment is type-compatible, and the var's type is inferred from the assignment.

## Template

```go
// loadBpfObjects is a stub replaced by bpf2go code generation.
// After running `go generate ./<package>/`, bpf2go produces
// <prefix>_bpfel.go containing the real implementation.
//
// This stub allows the package to compile before code generation.
// Callers should treat the returned error as a signal that the
// generated code is unavailable and operate in degraded mode.
var loadBpfObjects = func(objs *bpfObjects, opts *ebpf.CollectionOptions) error {
    return fmt.Errorf("<tool>: code generation not run — run 'go generate ./<package>/' with <toolchain requirements>")
}
```

## Key properties

1. **Compiles without the generated code.** The function body is a simple `return fmt.Errorf(...)` — no references to generated types beyond what the signature already declares.
2. **The generated code overrides the var at init time.** bpf2go generates an `init()` function or a file-level assignment that replaces the stub. The stub is never called when generation has been run.
3. **Type references in the package still resolve.** The `bpfObjects` struct and `ebpf.CollectionOptions` are declared in the same package (or imported), so Go can resolve them at compile time even though no real objects exist at init.
4. **Graceful degradation.** The caller wraps the load call in a best-effort pattern:

```go
func NewCollector(...) (*Collector, error) {
    c := &Collector{...}
    if err := c.loadEBPF(); err != nil {
        logger.Warn("ebpf unavailable, running in degraded mode", "err", err)
    }
    return c, nil
}
```

## Applicability

This pattern works for ANY code generation tool where:
- The generated code is optional (the system works without it, just with reduced functionality)
- The generated code produces Go types that are referenced in the same package
- The generated code includes a load/init function that can be stubbed

**Examples:** bpf2go (eBPF), protoc-gen-go (gRPC service stubs), stringer (enum String methods — less useful since these are usually required), go-bindata (embedded assets).

## Pitfalls

- **Don't use this for required generated code.** If the system CANNOT function without the generated code (e.g., protobuf message types that are fundamental to the API), the stub pattern is wrong — the package should fail to compile and the build system should enforce code generation.
- **The stub must match the generated function's signature.** If the generated function signature changes, update the stub. The `var` pattern is forgiving (Go infers), but a type mismatch still breaks compilation.
- **Don't let the stub hide real bugs.** If the generated code IS available but the stub is being called, check that `go generate` was run AND that the generated file is in the right package.

## Proven

Rabbit-Hole 2026-07-12 — `internal/collector/ebpf.go` used this pattern for `loadBpfObjects`. Without it, the package wouldn't compile on machines without clang, libbpf, and kernel headers. The stub allowed 17 collector tests to pass, build+vet green, and guard to pass — all without running bpf2go. The real eBPF loading path is exercised later when the toolchain is available.
